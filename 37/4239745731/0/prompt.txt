I got this feedback from our changes and the pr in general. Cn you did on this:

```
Issues to fix

🔴 Unstage the watcher revert. The diff in src/bun/modules/worktree/worktree.service.ts + package.json + bun.lock swaps @parcel/watcher for node:fs.watch. This violates spec point 1, and the swap is functionally weaker:

fs.watch(path, { recursive: true }) is unreliable across platforms — on Linux it requires Node ≥20 (and isn't fully equivalent in Bun); macOS recursive has historical edge cases. @parcel/watcher was picked precisely to dodge this.
You lose the native ignore glob support; the JS-side shouldIgnoreFileEvent filter still fires for every event in node_modules (the kernel still wakes you up — you just discard).
Removing @parcel/watcher after adding it via the ask-first list is itself an ask-first reversal per CLAUDE.md.

If there was a real reason to drop it (install pain on Bun, native binding issue, etc.), that needs a discussion + ADR before unwinding the spec.
🟡 Dead type after staged revert. FileChangeEvent in worktree.types.ts becomes unreferenced once @parcel/watcher is gone. Either delete it or keep it for the optional worktreeFileChanged message you might add later.
🟡 4th file in the module folder. worktree.test.ts lives alongside the trio. CLAUDE.md says "exactly three files." The Testing section is still a placeholder, so it's a minor breach, but this is the kind of thing you flagged yourself in the PIL-20 quick-guide — worth deciding now where colocated tests live (alongside, or tests/ mirror tree) and noting it in CLAUDE.md.
🟡 createWorktree RPC doesn't attach a watcher; only createWorktreesForFeature does. That's defensible (watchers only make sense for DB-tracked active worktrees, not bare git worktree add), but the spec says "createWorktree starts the watcher." Either clarify the spec to say "tracked-worktree creation" or add a watcher hook in the single-create path too.
```
